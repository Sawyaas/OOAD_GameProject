First Attempt at Use cases

Only being used as reference to "How not to do Use cases"

Brief Use Cases


Use case 1.0:Create a new Commander
Primary Actor:Player
System under discussion:Turn Based Strategy Game “Last Whisper”
Level:Player goal
Success Scenario:
  1. Use case begins when player opens game for the first time.
  2. Player gets option to proceed or quit.
  3. Player makes selection.
  4. Player creates,names, selects control and saves Commander.
  5. Use case ends when player exits game.

Casual Use Cases


Use Case 1.1: Create a new commander
Primary Actor:Player
Secondary Actor: Commander
System under discussion:Turn Based Strategy Game “Last Whisper”.
Level: Player goal
  1. Use case begins when player opens game for the first time.
  2. Player is given the option to select New Game, load Game or Exit.
  3. Player selects New Game.
  4. Game presents create commander screen.
  5. Commander must be given a name(between 6 and 12 characters).
  6. System selects, names and saves Commander
  7. Commanders are saved.
  8. Use case terminates when ‘endgame’ is selected.

Casual Use Cases


Use Case 1.2:Create a new commander
Primary Actor: Player 1,Player 2
Secondary Actor:Commander
System under discussion: Turn Based Strategy Game “Last Whisper”.
Business Scope:The ability within the game to create a Commander as seen by the Players.
Level:Summary
Preconditions: Player must have game installed on their computer.
Success Scenario:
  1. Use case begins when player opens game for the first time.
  2. Player is given the option to select New Game, Load Game or Exit.
      When selecting New Game the Player is given the option to Create and Name their Commander.
      When selecting Load Game the Players current saved game will be loaded for continuation
      from where the Player left off.

What I Learned from the process above

Learning curve

From the above process I learned that ‘my’ understanding of having to have a Brief, casual and
fully dressed Use Case for every Use Case was not the correct way to work through them.
Precondition
What I learned from the above attempts was how to establish what a Precondition is, A
precondition is simply a starting point for the Use Case to begin at. Use case begins with
some conditions met).

Example of Use case with Precondition

Withdrawer cash Use Case 1.0:
Withdrawer cashCustomer
Primary Actor:
Precondition:Card details have been validated and cash is available to account holder.
  1. The system presents the customer with Withdrawal Amount Screen
  2. The customer enters Withdrawal Amount .
  3. The system checks the withdrawal amount is not greater than the amount
    in the customer's account.
  4. If the Withdrawal Amount is less than the amount being sought the system pays out the cash.
  5. The system updates the customer's account details.
  6. The system asks if another service is requested.
  7. The system resets.
